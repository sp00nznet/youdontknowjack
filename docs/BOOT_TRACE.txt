NID 0x24A1EA07 = sys_ppu_thread_create (imported via sysPrxForUser).
ps3recomp implements thread creation as a SYSCALL (sys_ppu_thread.c) but never
registers it as a sysPrxForUser NID handler -> unresolved. The CRT's thread /
static-init then walks an uninitialized object table at the heap base
(0x40000030+) and calls heap addresses as function pointers -> segfault.

CURRENT BLOCKER (the "HLE bridge" phase):
  Access violation writing guest stack addr 0x0FEFFB78 as a RAW host pointer,
  inside the HLE call tree of func_000691F0 (which calls cellSysmodule/cellGcm/
  etc. import stubs). Same bug class as cellGameBootCheck: HLE functions
  registered via the generic adapter receive GUEST addresses for pointer args
  but dereference them as host pointers. Each pointer-taking HLE function the
  game uses needs vm_base translation (flOw does this in hle_modules.cpp).

  Systemic alternative worth evaluating: map guest memory at its NATIVE host VA
  (VirtualAlloc at the guest addresses, vm_base = 0) so guest==host addresses
  and NO per-function translation is ever needed -- eliminates the whole class.

v5 (NATIVE VA MAPPING):
Reserve the guest address space at its OWN host VAs (boot_main.cpp: 3 regions
for image/TLS/stack, heap, thread-stacks) and set vm_base = 0, so a guest
address IS its host address. A vectored handler commits reserved pages on first
touch. Result: guest==host, so lifted code (vm_base+a) AND HLE C code (raw
*guest_ptr) reach the same bytes -- the ENTIRE "HLE deref'd a guest pointer as
a host pointer" crash class is gone. (Caveats: vm.h's !vm_base "ready" checks
became VM_READY=ppu_vm_size; vm_oob now also guards a<0x10000 so lifted null
derefs read 0 instead of faulting the unmapped low pages.)

Boot now runs MUCH further:

v6 (native VA EVALUATED, then REVERTED to calloc; graphics-init HLE wired):

Native VA mapping (v5) is elegant -- guest==host kills the pointer-translation
crash class -- but it does NOT work on Windows for this title:
  * The OS reserves the low 64 KB, so guest page 0 is unmappable. The game's
    null-object inits write to page 0 (valid RAM on PS3) -> AV.
  * VirtualAlloc can't reserve 0x60000000..0xD0000000 (err 487: DLLs/process
    heap occupy parts of it), yet the game writes there (e.g. 0xCFB7E320).
  * It also does NOT remove the per-function HLE work: guest memory is still
    big-endian, so HLE pointer-out funcs must use vm_write* (byte-swap) anyway.
Conclusion: the flat calloc VM (vm_base + addr) is required -- it maps the FULL
32-bit guest space uniformly. Reverted boot_main/vm_oob to calloc; KEPT every
real fix (TOC-save, bcctrl, threads, cellGameBootCheck, cellVideoOut,
_cellGcmInitBody bridge, cellGcmGetTiledPitchSize, the crash+backtrace handler).

Graphics-init HLE wired this round (correct under calloc):
  * cellVideoOut* registered (gen_hle_nids now includes cellVideoOut.c) +
    cellVideoOutGetResolutionAvailability added (returns "available").
  * _cellGcmInitBody (0x15BAE46B) bridged to cellGcmSetupContext with guest
    alloc + vm_write32 callbacks (ppu_sysprx.cpp).
  * cellGcmGetTiledPitchSize uses vm_write32 (translate + byte-swap + OOB).

===========================================================================
*** SPU EXECUTION is the gate (confirmed) -- groundwork laid ***
===========================================================================
Drilled the render-loop block to its root. The boot does its init across two
threads: the main thread creates 3 anonymous event queues + the "AsyncLoad"
worker, then blocks in sys_event_queue_receive(queue=1, timeout=0) FOREVER.
Nothing ever sys_event_port_sends; no sys_spu_thread_group_create. The AsyncLoad
worker does _cellGcmInitBody (GCM/RSX init) and then itself blocks in a
HOST-level wait inside an HLE lib (it evades every probe: no lv2-wait, no
read8/16/32/64 spin, no lwmutex contention, no timer_usleep, no thread_exit) --
i.e. it is waiting for SPU/SPURS work that never runs. cellSpurs is management-
only (no SPU execution), so the SPURS event the worker needs is never produced,
the worker never signals queue 1, and the main thread waits forever.

GROUNDWORK DONE: extract_spu_images.py pulled 22 embedded SPU ELF images from
the EBOOT (project/spu/, ~450 KB; gitignored). The runtime already has the SPU
execution pipeline (runtime/spu/spu_workload.c: fingerprint -> pre-lifted native
entry -> 256KB local store -> run; spu_channels/spu_dma; tools/spu_lifter.py).
Added a SPURS-completion shim (cellSpursEventFlagWait now times out and
force-satisfies after a grace period) as a stopgap for when a SPURS event flag
IS the awaited object.

NEXT: the real fix is SPU EXECUTION -- lift the title's SPU binaries
(spu_disasm/spu_lifter), register them (spu_workload_register), and wire

and produce the events the PPU side waits on. This is a major subsystem (the
hardest part of any PS3 port) and the clear next milestone.
