PROGRESS TIMELINE
-----------------
v1 (literal import stubs):
  entry dispatched, TLS, 170MB heap alloc -> FATAL stuck calling 0x39800000
  Root cause: the lifter emitted the LITERAL .lib.stub trampolines, which deref
  an import pointer table the recomp never fills -> bctrl into the bytes of
  'li r12,0' (0x39800000). This is the wall gunstar is also stuck on.

v2 (lifter --hle-stubs : each of the 265 import stubs -> ps3_hle_call(nid)):
  PAST the 0x39800000 wall. CRT now runs much deeper:
    [crt] sys_initialize_tls: block 0x0E000000, r13=0x0E007000
    [sys_memory] allocate(0xAA00000) -> 0x40000000        (170MB heap)
    [hle] unresolved NID 0x24A1EA07                        <-- next blocker
    [ppu] unresolved indirect call -> 0x40000030 .. 0x40000120  (vtable sweep)
    -> segfault during static-constructor / vtable init

NEXT BLOCKER
------------
NID 0x24A1EA07 = sys_ppu_thread_create (imported via sysPrxForUser).
ps3recomp implements thread creation as a SYSCALL (sys_ppu_thread.c) but never
registers it as a sysPrxForUser NID handler -> unresolved. The CRT's thread /
static-init then walks an uninitialized object table at the heap base
(0x40000030+) and calls heap addresses as function pointers -> segfault.

v3 (sys_ppu_thread_create + thread entry trampoline wired):
  NID 0x24A1EA07 now dispatches to the real lv2 thread create; the CRT spawns
  its "AsyncLoad" worker (entry OPD 0x530E38) and the new generic thread entry
  trampoline (ppu_loader.cpp: resolve OPD -> {code,toc}, share main TLS,

NEXT BLOCKER (main thread, static init):
  Before/independent of the thread, the main thread's C++ static-constructor
  dispatch is broken: it makes indirect calls to small ints (0x2, 0x3E8=1000)
  and sweeps a heap table (0x40000120..0x40000030, 8-byte stride) calling each
  as a function -> all unresolved -> then a host segfault. These are VIRTUAL
  calls through objects whose vtables were never initialised, because the
  global-constructor (__init_array) walk produced heap addresses instead of
  constructor OPD pointers. Suspect the CRT table-copy loops in func_0005EDFC
  (loc_0005EE60 / loc_0005EEA4) or an r2(TOC)-relative base the lifter mishandles.
  TODO: trace which guest store fills 0x40000030.. and what the init_array
  source pointer should be; this is the deep-CRT static-init bring-up step.

---------------------------------------------------------------------------
v4 (TOC-save + bcctrl + thread stacks + cellGame fixes):

Four correctness fixes cleared the static-init corruption and got the boot into
real game init:
  * ppu_hle: save caller TOC r2 to 0x28(r1) on every import call (PPC64 ELFv1).
    The --hle-stubs body had dropped it -> garbage r2 after imports -> the whole
    0x40000030 vtable sweep + 40 OOB accesses. FIXED: those are gone.
  * ppu_lifter: bcctrl / b<cond>ctrl now route to ps3_indirect_call (was
    trampolining 5,596 vtable calls to a garbage func_00000030).
  * boot harness: VM grown to 0xE0000000 so thread stacks at 0xD0000000 are
    backed; spawned threads + main get 256 MB host stacks (deep recomp nesting).
  * cellGameBootCheck: translate its out-pointers through vm_base + vm_write32
    (was writing the guest stack address as a raw host pointer -> AV).

Boot now: TLS -> heap -> AsyncLoad thread spawns & runs -> cellGame BootCheck ->
into game init. A last-chance crash handler (boot_main.cpp) prints the faulting
address + RIP rva for llvm-symbolizer.

CURRENT BLOCKER (the "HLE bridge" phase):
  Access violation writing guest stack addr 0x0FEFFB78 as a RAW host pointer,
  inside the HLE call tree of func_000691F0 (which calls cellSysmodule/cellGcm/
  etc. import stubs). Same bug class as cellGameBootCheck: HLE functions
  registered via the generic adapter receive GUEST addresses for pointer args
  but dereference them as host pointers. Each pointer-taking HLE function the
  game uses needs vm_base translation (flOw does this in hle_modules.cpp).

  Systemic alternative worth evaluating: map guest memory at its NATIVE host VA
  (VirtualAlloc at the guest addresses, vm_base = 0) so guest==host addresses
  and NO per-function translation is ever needed -- eliminates the whole class.
