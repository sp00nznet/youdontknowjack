*** Two PPC/x86 lifter bugs fixed -> boot reaches thread-pool spin-up ***
===========================================================================
Watchdog thread-RIP snapshots pinned the post-cellAudioQuit hang to an exact
lifted function, which exposed two real lifter correctness bugs:

 1. PPC rA=0 addressing. In indexed/reservation addressing an rA field of 0
    means the LITERAL 0, not GPR r0. func_000DE9E0 is the canonical atomic-incr
    loop (lwarx rD,0,r3; add r0,...; stwcx. rS,0,r3) -- it reuses r0 as scratch,
    so emitting ctx->gpr[0] made the lwarx and stwcx. EAs diverge, the
    reservation never matched, and the AsyncLoad worker spun forever. Fixed in
    lwarx/stwcx./ldarx/stdcx. and all non-update X-form memory ops (byte-reverse,
    dcbz, AltiVec) via a shared _xea() helper.

 2. PPC non-trapping divide. divw/divwu/divd don't trap on /0 or INT_MIN/-1
    (undefined result, no fault); the host C divide raised SIGFPE (0xC0000094)
    in func_00263194. Guarded to yield 0, matching hardware.

With both fixed the worker runs FAR deeper: past cellAudioQuit, past the divide,
and the title now spawns its WORKER-THREAD POOL (6+ threads) -- previously only
main + AsyncLoad existed. New frontier: a couple of those workers sit in
func_00063428 and the AsyncLoad worker touches a NULL-relative address
(0xFFFFFExx -> vm_oob); the synthetic GCM vblank/flip handler OPDs
(0x00530D70/78) also resolve to non-4-aligned garbage code pointers and want
looking at. The SPU dispatch pipeline (built, 22 images) is still downstream of
this, waiting for the boot to reach the first cellSpursCreateTask/AddWorkload.

*** ROOT-CAUSE FIX: callee-saved-register frame-drift (OOB eliminated) ***
===========================================================================
Confirmed and fixed the corruption that caused the boot hang -- a whole CLASS
of frame-drift bugs, not a one-off.

PROVEN MECHANISM (with runtime probes): func_00182DD8 saves a small loop count
(r29 = 8) to its frame at sp+0x78; a callee whose stack pointer has drifted up
(a merged-function tail-call running an epilogue against the wrong frame) writes
the TOC pointer into that exact slot; func_00182DD8's epilogue reloads
0x00544370 instead of 8 -> func_003DC468 loops 5.5M times -> hang. Before/after
probes showed r29 = 0x8 before the call, 0x00544370 after.

FIX (ppu_lifter.py, lift_function): the PPC ABI guarantees r14-r31 survive a
call. Instead of reloading them from the (callee-corruptible) frame slot at the
epilogue, snapshot each saved non-volatile GPR at function ENTRY into a C local
and restore from that. Only restores that PAIR with a save (same offset+reg) in
the same body are rewritten -- a mid-function tail-entry has the restore but not
the prologue save, so its entry snapshot would be the mid-function value, not the
caller's; those are left alone (rewriting them reintroduced the OOB -> reverted).
42,869 restores rewritten. RESULT: OOB accesses 0 (was a runaway loop); the
r29-count loop is gone.

REMAINING (resume here): func_003DC468 still spins in a SECOND loop (no OOB),
reached via func_002963F4 -> func_00264D58 -> func_002965B4 -> func_0026574C ->
func_00264A4C -> func_003DC468 (recovered with the new watchdog stack-scan). Its
loop limit is another callee-saved reg (gpr[26]); the residual corruption is the
SAME class one level deeper -- an UNPROTECTED tail-entry callee (no matching
save in its body, so the pairing fix skips it) clobbers a callee-saved register
in the shared frame. The clean fixes:
  (a) eliminate the merged functions / problematic tail-entries via better
      boundary detection so there are no shared-frame tail-entries, OR
  (b) give tail-entries a correct snapshot (they can't capture the caller's
      saved reg from their own entry, so this needs the original function's
      prologue value threaded in -- non-trivial).
Boundary recovery currently lands 4379 splits but ~452 functions stay merged
(mflr-first / entry-quirk prologue patterns it doesn't yet match); closing those
removes the tail-entries that still corrupt.

*** ROOT CAUSE: jump-table dispatcher missed (lwzx operand order) ***
===========================================================================
The func_003DC468 loc_003DC548 "linked-list walk" hang was NOT a circular list
or a frame-drift remnant. Tracing the loop registers (runtime probe) showed:
  loop limit  r8  = 0x00000000        (should be a hash-table element count)
  list ptr    r10 = 0x04000008        (constant; never advances)
  iterations  -> 1,000,000+           (infinite)
r10 derives from r7 = *(r29+0x14) = 0x04000000, a hash-table pointer into
UNALLOCATED guest memory (flat VM backs it but it reads as 0), so r8 = hash &
*(table+4) = hash & 0 = 0 -> bucket 0 -> degenerate chain -> spin.

WHY the table is zeroed: just before the hang the runtime logs
  [ppu] unresolved indirect call -> 0x002965E8
  [ppu] unresolved indirect call -> 0x0029664C
ps3_indirect_call NO-OPS on unresolved targets, so the init that builds the
hash table never runs. 0x002965E8 / 0x0029664C are NOT functions -- they are
CASE blocks of a gcc switch (an unrolled / Duff's-device remainder handler;
each sets N 2-bit fields then `b 0x2969DC`). The dispatcher (func_002963F4,
@0x296590) is a textbook offset jump table:
  cmplwi cr7,r3,0x7 ; lwz r9,-0x4D70(r2) ; lwzx r0,r9,r0 ; add r0,r0,r9 ;
  mtctr r0 ; bctr        (table embedded in .text at 0x296594)

discover_jump_tables() SHOULD have recovered this, but had a real bug: for
`lwzx rD, rA, rB` it hardcoded the table-base register as rB (p[2]). Here the
base is rA (r9, the TOC-loaded pointer) and rB (r0) is the index. So the
`lwz base, disp(r2)` probe looked for the wrong register, found nothing, set
disp=None, and SKIPPED the dispatcher entirely -> the 4 case blocks were never
lifted -> the unresolved indirect calls -> skipped init -> the zeroed table.

FIX (tools/ppu_lifter.py discover_jump_tables): try BOTH lwzx operands as the
candidate base; the real base is whichever is loaded TOC-relative via
`lwz base, disp(r2)`. Verified against the ELF: recovers exactly
0x2965B4 / 0x2965E8 / 0x29664C / 0x2966DC. This is a GENERAL fix -- every
dispatcher with the operands swapped was being skipped binary-wide, so it may
clear several latent hangs at once.
NEXT: re-lift, confirm the unresolved indirect calls are gone and the
func_003DC468 hash-walk terminates, then see how much deeper the boot reaches.
