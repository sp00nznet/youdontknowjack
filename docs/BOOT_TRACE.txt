*** Two PPC/x86 lifter bugs fixed -> boot reaches thread-pool spin-up ***
===========================================================================
Watchdog thread-RIP snapshots pinned the post-cellAudioQuit hang to an exact
lifted function, which exposed two real lifter correctness bugs:

 1. PPC rA=0 addressing. In indexed/reservation addressing an rA field of 0
    means the LITERAL 0, not GPR r0. func_000DE9E0 is the canonical atomic-incr
    loop (lwarx rD,0,r3; add r0,...; stwcx. rS,0,r3) -- it reuses r0 as scratch,
    so emitting ctx->gpr[0] made the lwarx and stwcx. EAs diverge, the
    reservation never matched, and the AsyncLoad worker spun forever. Fixed in
    lwarx/stwcx./ldarx/stdcx. and all non-update X-form memory ops (byte-reverse,
    dcbz, AltiVec) via a shared _xea() helper.

 2. PPC non-trapping divide. divw/divwu/divd don't trap on /0 or INT_MIN/-1
    (undefined result, no fault); the host C divide raised SIGFPE (0xC0000094)
    in func_00263194. Guarded to yield 0, matching hardware.

With both fixed the worker runs FAR deeper: past cellAudioQuit, past the divide,
and the title now spawns its WORKER-THREAD POOL (6+ threads) -- previously only
main + AsyncLoad existed. New frontier: a couple of those workers sit in
func_00063428 and the AsyncLoad worker touches a NULL-relative address
(0xFFFFFExx -> vm_oob); the synthetic GCM vblank/flip handler OPDs
(0x00530D70/78) also resolve to non-4-aligned garbage code pointers and want
looking at. The SPU dispatch pipeline (built, 22 images) is still downstream of
this, waiting for the boot to reach the first cellSpursCreateTask/AddWorkload.

*** ROOT-CAUSE FIX: callee-saved-register frame-drift (OOB eliminated) ***
===========================================================================
Confirmed and fixed the corruption that caused the boot hang -- a whole CLASS
of frame-drift bugs, not a one-off.

PROVEN MECHANISM (with runtime probes): func_00182DD8 saves a small loop count
(r29 = 8) to its frame at sp+0x78; a callee whose stack pointer has drifted up
(a merged-function tail-call running an epilogue against the wrong frame) writes
the TOC pointer into that exact slot; func_00182DD8's epilogue reloads
0x00544370 instead of 8 -> func_003DC468 loops 5.5M times -> hang. Before/after
probes showed r29 = 0x8 before the call, 0x00544370 after.

FIX (ppu_lifter.py, lift_function): the PPC ABI guarantees r14-r31 survive a
call. Instead of reloading them from the (callee-corruptible) frame slot at the
epilogue, snapshot each saved non-volatile GPR at function ENTRY into a C local
and restore from that. Only restores that PAIR with a save (same offset+reg) in
the same body are rewritten -- a mid-function tail-entry has the restore but not
the prologue save, so its entry snapshot would be the mid-function value, not the
caller's; those are left alone (rewriting them reintroduced the OOB -> reverted).
42,869 restores rewritten. RESULT: OOB accesses 0 (was a runaway loop); the
r29-count loop is gone.

REMAINING (resume here): func_003DC468 still spins in a SECOND loop (no OOB),
reached via func_002963F4 -> func_00264D58 -> func_002965B4 -> func_0026574C ->
func_00264A4C -> func_003DC468 (recovered with the new watchdog stack-scan). Its
loop limit is another callee-saved reg (gpr[26]); the residual corruption is the
SAME class one level deeper -- an UNPROTECTED tail-entry callee (no matching
save in its body, so the pairing fix skips it) clobbers a callee-saved register
in the shared frame. The clean fixes:
  (a) eliminate the merged functions / problematic tail-entries via better
      boundary detection so there are no shared-frame tail-entries, OR
  (b) give tail-entries a correct snapshot (they can't capture the caller's
      saved reg from their own entry, so this needs the original function's
      prologue value threaded in -- non-trivial).
Boundary recovery currently lands 4379 splits but ~452 functions stay merged
(mflr-first / entry-quirk prologue patterns it doesn't yet match); closing those
removes the tail-entries that still corrupt.

*** ROOT CAUSE: jump-table dispatcher missed (lwzx operand order) ***
===========================================================================
The func_003DC468 loc_003DC548 "linked-list walk" hang was NOT a circular list
or a frame-drift remnant. Tracing the loop registers (runtime probe) showed:
  loop limit  r8  = 0x00000000        (should be a hash-table element count)
  list ptr    r10 = 0x04000008        (constant; never advances)
  iterations  -> 1,000,000+           (infinite)
r10 derives from r7 = *(r29+0x14) = 0x04000000, a hash-table pointer into
UNALLOCATED guest memory (flat VM backs it but it reads as 0), so r8 = hash &
*(table+4) = hash & 0 = 0 -> bucket 0 -> degenerate chain -> spin.

WHY the table is zeroed: just before the hang the runtime logs
  [ppu] unresolved indirect call -> 0x002965E8
  [ppu] unresolved indirect call -> 0x0029664C
ps3_indirect_call NO-OPS on unresolved targets, so the init that builds the
hash table never runs. 0x002965E8 / 0x0029664C are NOT functions -- they are
CASE blocks of a gcc switch (an unrolled / Duff's-device remainder handler;
each sets N 2-bit fields then `b 0x2969DC`). The dispatcher (func_002963F4,
@0x296590) is a textbook offset jump table:
  cmplwi cr7,r3,0x7 ; lwz r9,-0x4D70(r2) ; lwzx r0,r9,r0 ; add r0,r0,r9 ;
  mtctr r0 ; bctr        (table embedded in .text at 0x296594)

discover_jump_tables() SHOULD have recovered this, but had a real bug: for
`lwzx rD, rA, rB` it hardcoded the table-base register as rB (p[2]). Here the
base is rA (r9, the TOC-loaded pointer) and rB (r0) is the index. So the
`lwz base, disp(r2)` probe looked for the wrong register, found nothing, set
disp=None, and SKIPPED the dispatcher entirely -> the 4 case blocks were never
lifted -> the unresolved indirect calls -> skipped init -> the zeroed table.

FIX (tools/ppu_lifter.py discover_jump_tables): try BOTH lwzx operands as the
candidate base; the real base is whichever is loaded TOC-relative via
`lwz base, disp(r2)`. Verified against the ELF: recovers exactly
0x2965B4 / 0x2965E8 / 0x29664C / 0x2966DC. This is a GENERAL fix -- every
dispatcher with the operands swapped was being skipped binary-wide, so it may
clear several latent hangs at once.
NEXT: re-lift, confirm the unresolved indirect calls are gone and the
func_003DC468 hash-walk terminates, then see how much deeper the boot reaches.

SPU FRONTIER, root cause traced to the exact instruction (architectural)
---------------------------------------------------------------------------
Drilled the persistent SPU job (image 22 = spu_0021_at_004E5F80, entry
spu_func_00003050) all the way down. Channel trace showed it does ZERO channel
ops before hanging -> it is NOT blocked on DMA/mailbox; it spins in pure
computation. Per-callee trace pinned the hang exactly:
  entry 3050 -> func_00003010 (returns) -> func_00003E68 (NEVER returns)
func_00003E68 is a prologue that does:
    gpr[3] = rotmai(gpr[3], 112)          ; (arith shift of the task arg)
    if (gpr[3] == 64) -> func_00003ED8    ; real path: MFC DMA of the context
    else              -> func_00003ED0    ; func_00003ED0 is `nop; goto self`
func_00003ED0 is an INFINITE HALT-LOOP (the SPU task-runtime's abort/assert
path). The arg check fails (gpr[3] derives from args_ea=0x0FEFF480 -> >>16 =
0x0FEF, not 64), so the runtime aborts into the halt spin.

ARCHITECTURAL ROOT (why the check fails): image 22 is a SPURS *task-runtime*
image. On real hardware the SPURS *kernel* (policy module, a SEPARATE SPU image
loaded by cellSpursInitialize) sets up each task's SPU execution context --
registers, LS layout, the DMA'd task descriptor -- BEFORE jumping to the task
runtime entry, which then validates it (the gpr[3]==64 assert). Our HLE
cellSpursInitialize is a STUB: the real SPURS kernel never runs. So
cellSpursCreateTask dispatches the task image in ISOLATION with only r3=args_ea,
the runtime's context assert fails, and it halts. Downstream, the PPU busy-polls
sys_event_flag_wait(flag=1000,bits=0) forever because no task ever runs/signals.

THE REAL FIX (large, the next milestone): emulate SPURS properly -- run the
actual SPURS kernel image on the SPU worker threads (from cellSpursInitialize),
have it consume the workload/taskset queues the game builds in main memory, and
let IT dispatch tasks with the correct per-task SPU context + signal completion
back to the lv2 event flag/queue the PPU waits on. That replaces the current
"CreateTask -> dispatch the task image directly" shortcut, which can't satisfy
the task runtime's context contract. Only then does the SPURS handshake complete
and the game advance toward its first cellGcmSetFlip (then: framebuffer EA fix
for AddressToOffset(0), and the rsx_d3d12_backend.c present path -> pixels).

SPU halt is REAL binary (not a lifter bug) — exact assert + ruled-out shortcuts
---------------------------------------------------------------------------
Disassembled the SPU ELF (spu_disasm) at the halt to rule out a mis-lift.
LS 0x3000 maps to file 0x100; LS 0x3EC0 region is, verbatim in the binary:
    3ECC: brnz $r2, 0x3ED8        ; take real path iff r2 != 0
    3ED0: nop
    3ED4: br   0x3ED0             ; genuine infinite halt-loop (NOT a lifter bug)
    3ED8: ...real path (MFC DMA)...
and the condition (func_00003E68) is, verbatim:
    3E70: rotmai $r3, $r3, 112    ; r3 = arg >> 16  (arithmetic)
    3E88: ceqi   $r2, $r3, 64     ; r2 = (arg>>16 == 64)
So the SPU task runtime requires its r3 argument to satisfy (r3 >> 16) == 64,
i.e. r3 in [0x00400000, 0x0040FFFF].

Logged every candidate guest EA we could pass as r3:
    taskset_ea = 0x4005E000  (>>16 = 0x4005)
    spurs_ea   = 0x40009D00  (>>16 = 0x4000)
    elf_ea     = 0x004F5F80  (>>16 = 0x004F)
    ctx_ea     = 0x0FEFF480  (>>16 = 0x0FEF)
NONE is 0x0040xxxx. So the required r3 is NOT any pointer the HLE has -- it is a
value the SPURS KERNEL constructs when it dispatches a task (a packed
magic<<16 | index, or a pointer into a SPURS-reserved region). This conclusively
rules out the quick fixes (swap which EA we pass as r3; patch the lifter) and
confirms the only correct path is implementing the SPURS kernel's task-dispatch
ABI: run the SPURS policy module, have it pull tasks from the taskset queue and
invoke each task image with the kernel-constructed r3 + LS/context, then signal
completion to the lv2 event flag/queue (flag 1000) the PPU busy-polls. That is
the next milestone before the game advances to cellGcmSetFlip and pixels.

SPU DISPATCH ABI — REVERSE-ENGINEERED (validated empirically)
---------------------------------------------------------------------------
Made concrete progress decoding the SPURS task-dispatch ABI by experiment
(temporarily forcing a passing r3 + logging every MFC DMA, then reverted):

1. r3 is NOT a scalar EA — it is a 128-bit {handle, context_EA} pair:
     r3.word0 = handle; its HIGH halfword must == 0x40 (the assert
                rotmai r3,112 == 64, i.e. (word0>>16)==0x40 -- a type tag).
     r3.word1 = the effective address of the task/kernel CONTEXT.
   Proven: forcing r3.word0=0x00400000 passes the assert; then forcing
   r3.word1=0x12340000 made the runtime's first DMA read exactly
   `GET lsa=0x32500 ea=0x12340000 size=64` (hand-derived through the quadword
   shuffle `rotqbyi r9,4; rotqmbyi …,124; shlqbyi …,4` and confirmed at runtime).
   r4 does NOT feed the EA (tested: r4=0x12340000 changed nothing).

2. The runtime then GETs a 64-byte context from r3.word1 into LS 0x32500,
   reads pointer fields out of it (e.g. lqr from LS 0x32510/0x32530), and issues
   further DMAs (128 bytes -> LS 0x32580, etc.) from EAs stored IN that context.
   With an empty/zero context it loops re-reading EA 0 forever (the spin we see).

NEXT CONCRETE STEP (continue the RE): map the 64-byte context layout — trace
which LS 0x32500+offset words the runtime loads and how it uses them (they are
the EAs for the 128-byte GET etc.), then build a matching context in guest mem,
set r3 = {0x0040xxxx handle, context_EA}, and iterate until the runtime stops
spinning and runs the task body. This is the foundation of the HLE SPURS kernel
(cellSpursInitialize must allocate the context region + the kernel must encode
this r3 per task and signal completion to lv2 event flag 1000 on task end).

SPU context is a LOCK-FREE QUEUE — image 22 is the SPURS kernel (refined)
---------------------------------------------------------------------------
Continued the RE: pointed r3.word1 at a controlled region filled with per-word
markers (0xC0DE00<off>) + logged DMAs. Findings:
- The runtime GETs 64 bytes from r3.word1, MODIFIES, and PUTs 64 bytes back

  static descriptor read. The marker words do NOT pass through as EAs; the next

  context it spins. (Also a `PUT size=1 ea=1` -- a byte flag write.)

  the scheduler), and r3.word1 is the SPURS instance's lock-free WORKLOAD-QUEUE
  context. The kernel should be started by cellSpursInitialize (one per SPU),
  poll that queue, claim workloads/tasks the PPU enqueues, run them, and signal
  completion. Our cellSpursCreateTask dispatching the image directly is the wrong
  shape; and cellSpursInitialize (a stub) never builds the queue, so the claim
  always fails.
